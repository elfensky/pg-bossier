# Goal 3 â€” Retry history lineage edge cases (design)

**Status:** v2 â€” post-adversarial-review.
**Tracking issue:** [#4](https://github.com/elfensky/pg-bossier/issues/4).
**Charter rubric:** Goal 3 of [issue #1](https://github.com/elfensky/pg-bossier/issues/1).
**Adversarial review:** [`docs/superpowers/debates/2026-05-24-goal-3-retry-history-lineage/`](../debates/2026-05-24-goal-3-retry-history-lineage/) â€” three reviewers (Gemini, Sonnet, Opus), two rounds. Codex CLI was unavailable. Convergence on a clear set of changes; this v2 incorporates them.

## What ships

Two methods on the existing `bossier` client, one targeted fix to Goal 2's existing writer, and four documentation updates. **No new schema column. No new pg-boss surface touched.** The capture trigger is unchanged.

1. **`bossier.recordDeadLetter({ sourceJobId, dlqJobId })`** â€” an opt-in JS writer that records the sourceâ†’DLQ relationship in `terminal_detail` JSONB on the source's last `failed` attempt row.
2. **`bossier.findDeadLetterSource(dlqJobId)`** â€” the reverse-lookup reader. Returns the source job's `{ jobId, attempt, queue }` or `null`.
3. **`bossier.findDeadLetterTarget(sourceJobId)`** â€” forward-direction reader. Returns the DLQ job's id or `null`. (Gemini's R1 proposal; Sonnet R2 conceded as should-land.)
4. **Fix to Goal 2's `recordTerminalDetail` SQL.** The current `SET terminal_detail = $4::jsonb` is changed to `SET terminal_detail = COALESCE(terminal_detail, '{}'::jsonb) || $4::jsonb`. This is the **OPTION-A** fix from the adversarial review and is required to make this whole composition correct. Without it, `recordTerminalDetail` silently wipes `deadLetteredAs` (and any other key written via JSONB merge). With it, both writers cooperate at the key level.
5. **Documentation updates** â€” README "Recording dead-letter lineage" section, COMPATIBILITY.md confirmation that no new pg-boss surface is added, CHANGELOG entry under `[Unreleased]`, CLAUDE.md project-status + goal-status table sync.

## Why

pg-boss's dead-letter routing is the *only* path in pg-boss 12 that produces a genuinely new job id with no link back to the source job. From pg-bossier's chronicle, a forensic query against a DLQ job id cannot reach back to the original job's attempts, inputs, or failure history. descent-app's ~45 raw-SQL queries against `pgboss.*` ([descent-app#343](https://github.com/drunikbe/descent-app/issues/343)) include the "what original job is this DLQ entry?" pattern; closing this gap with a typed call lets that query disappear. The v1 success criterion from issue #1 â€” *"'what happened to job X six months ago?' is answerable with one typed query â€” including inputs, final output, failure class, and full retry history"* â€” applies to DLQ jobs too.

## Decisions locked (post-debate)

### 1. Detection is impossible at the trigger

**3-of-3 verified independently.** Sonnet read `node_modules/pg-boss/dist/plans.js` `failJobs()` directly (lines 1029â€“1189):

- `failed_jobs` CTE inserts the source's new `state='failed'` row. The capture trigger fires here.
- `dlq_jobs` CTE inserts the DLQ row in the SAME statement, AFTER `failed_jobs` by CTE data dependency.
- DLQ row's UUID is generated by `gen_random_uuid()` DEFAULT â€” not knowable at the failed-row trigger firing time.

Three escape hatches were checked (statement-level trigger with transition tables, deferred constraint triggers, session state via `set_config`). All collapse on the same correlation problem under concurrency. Schema-column options (`dead_letter_source_id`) and link-table options collapse on this fact. They can't be reliably populated.

### 2. Application-layer writer, called from the DLQ-handler side

**3-of-3.** The DLQ-handler has both the source `jobId` (from its own bookkeeping â€” see `_originalJobId` contract below) and the new `dlqJobId` (the job it just received). It's the only place that has BOTH at the same time. A focused sibling method (`recordDeadLetter`, not an extension to `recordTerminalDetail`) keeps the call site clean: it doesn't require `attempt` or a valid `class`-bearing payload (both of which would kill adoption per the prior debate's tiebreaker).

### 3. OPTION-A: Fix Goal 2's writer to use JSONB merge

**2-of-3 explicit (Gemini, Sonnet); Opus R2 conceded after Sonnet's correction.** Goal 2's shipped `recordTerminalDetail` does a blind `SET terminal_detail = $4::jsonb`. The spec v1 claimed associative merge â€” false against the code. The fix is a two-line SQL change in `src/terminal-detail.ts:100`:

```sql
SET terminal_detail = COALESCE(terminal_detail, '{}'::jsonb) || $4::jsonb
```

**What changes in Goal 2's semantic:** `recordTerminalDetail` becomes a *key-level* writer instead of a *full-object* writer. Concrete behavior:

- `recordTerminalDetail({state: 'failed', detail: {class: 'transient', message: 'attempt 1'}})` followed by `recordTerminalDetail({state: 'failed', detail: {message: 'attempt 2'}})` â†’ row's `terminal_detail = {class: 'transient', message: 'attempt 2'}`. The `class` key from the first call is PRESERVED; the `message` key is overwritten.
- `recordTerminalDetail({class: 'non_retryable'})` followed by `recordTerminalDetail({class: 'transient'})` â†’ row's `terminal_detail.class = 'transient'`. Same-key overwrite still works.

This is strictly better than the v1 behavior. A second call to `recordTerminalDetail` on the same row is either a bug (wrong jobId/attempt) or deliberate enrichment (worker wants to add fields). In neither case should the second write silently destroy the first. Goal 2's existing "concurrent calls â€” last-writer-wins" test continues to hold *at the key level* (which is the relevant level for a JSONB audit field).

OPTION-B (mandate ordering across processes) and OPTION-C (separate column) were both rejected after R2 debate.

### 4. Cardinality is 1:1, not plural

**3-of-3 after R2.** pg-boss 12's `failJobs` CTE emits exactly one DLQ INSERT per final failure. The "multiple DLQ ids per source" scenario is structurally handled by the per-attempt row model in `pgbossier.record` â€” each retry attempt is its own row that can carry its own `deadLetteredAs`. `deadLetteredAs: string` (single), not `string[]`.

### 5. Reverse-lookup reader ships in the same PR; forward reader as should-land

**Reverse reader:** `findDeadLetterSource` is must-land. Writer-without-reader is the worst shape.

**Forward reader:** `findDeadLetterTarget` is should-land. Two lines of SQL; logical mirror of the reverse reader. Sonnet R2 conceded it belongs in v1.

### 6. Singleton supersession â€” nothing to model

**4-of-4 unanimous in the prior debate.** No pg-boss 12 mechanism produces a displaced job. `ON CONFLICT DO NOTHING` on the singleton unique indexes drops the newcomer; the incumbent stays. Resolved by absence.

### 7. Reschedule â€” same row-version, no marker

**4-of-4 unanimous in the prior debate.** A reschedule is an UPDATE that resets `start_after`. Capture trigger picks it up as an ordinary row-version of the same id.

### 8. Decisions explicitly punted (separate issues)

- `singleton_key` capture as a plain column on `pgbossier.record` â€” Sonnet's observational improvement; separate issue.
- Skipping housekeeping-only UPDATEs in the trigger â€” under #21 perf follow-ups.
- Reserved data-key convention (`_pgbossier_source_id`) â€” rejected 3-of-4 in the prior debate.
- Statement-level after-trigger correlation â€” verified impossible; not even a future option without pg-boss changes.
- Lineage-integrity diagnostic (find DLQ rows without source link) â€” useful ops tool, follow-up issue.
- Chained-DLQ traversal helper (walk full Aâ†’Bâ†’C in one call) â€” nice-to-have, defer.

## Consumer contract â€” `_originalJobId` (load-bearing)

**`recordDeadLetter` requires the DLQ-handler to know the source `jobId`.** pg-bossier does not derive this â€” the substrate impossibility (decision 1) means we have no way to recover it from pg-boss state. The consumer is responsible for preserving the source id by setting it on the source job's `data` payload BEFORE calling `boss.send()`. Convention:

```js
await boss.send('image-processing', {
  _originalJobId: crypto.randomUUID(),  // self-identifying field
  ...payload
});
```

Then in the DLQ handler:

```js
boss.work('failures-to-review', async (job) => {
  await bossier.recordDeadLetter({
    sourceJobId: job.data._originalJobId,
    dlqJobId: job.id,
  });
  // ... handler-specific recovery logic ...
});
```

Without this discipline, `recordDeadLetter` cannot be called correctly. This is a **named, surface-level requirement** â€” not a buried convention. It belongs in the README's "Recording dead-letter lineage" section, in the JSDoc on `recordDeadLetter`, and in the spec's success criteria below.

(Alternative: pg-bossier could provide a `bossier.send()` wrapper that auto-injects `_originalJobId`. Out of scope for v1; named here as a future ergonomic improvement.)

## API surface

### Writer

```typescript
client.recordDeadLetter({
  sourceJobId: string;  // the failed source job's id (from consumer bookkeeping)
  dlqJobId: string;     // the new DLQ job's id (from job.id in the DLQ handler)
}): Promise<void>;
```

Behavior:
- Finds the most recent `state = 'failed'` row in `pgbossier.record` for `sourceJobId` (`ORDER BY attempt DESC LIMIT 1`).
- Performs a JSONB merge: `terminal_detail = COALESCE(terminal_detail, '{}'::jsonb) || jsonb_build_object('deadLetteredAs', $2::text)`.
- **Idempotent at the value level.** Calling twice with the same `(sourceJobId, dlqJobId)` produces the same merged value (SQL still runs the UPDATE).
- **Conflicting second call** (same `sourceJobId`, *different* `dlqJobId`) â†’ logs a warning and is a no-op. The first link wins. This prevents consumer-bug silent corruption (e.g., a DLQ-handler that retries with a freshly-generated DLQ id).
- **Fail-open.** Logs and continues on any DB error.
- **Silent no-op** when the source row is not found (wrong id, source row purged, source job never reached `failed`). Logged at WARNING with a `reason: 'not_found'` field so ops can distinguish from `reason: 'db_error'`.

SQL (sketch â€” actual implementation handles the conflict check):

```sql
WITH target AS (
  SELECT job_id, attempt, terminal_detail
  FROM pgbossier.record
  WHERE job_id = $1 AND state = 'failed'
  ORDER BY attempt DESC
  LIMIT 1
), should_write AS (
  -- Only write if no existing deadLetteredAs OR existing value matches the new dlqJobId.
  SELECT job_id, attempt FROM target
  WHERE terminal_detail IS NULL
     OR NOT (terminal_detail ? 'deadLetteredAs')
     OR terminal_detail->>'deadLetteredAs' = $2
)
UPDATE pgbossier.record r
SET terminal_detail = COALESCE(r.terminal_detail, '{}'::jsonb)
                    || jsonb_build_object('deadLetteredAs', $2::text)
FROM should_write w
WHERE r.job_id = w.job_id AND r.attempt = w.attempt;
```

Cost: one indexed point lookup + one row update. Well under the published ~1ms per-event budget.

### Reverse reader

```typescript
client.findDeadLetterSource(
  dlqJobId: string
): Promise<{ jobId: string; attempt: number; queue: string } | null>;
```

Behavior:
- Queries `pgbossier.record` for the row whose `terminal_detail` contains `{ deadLetteredAs: <dlqJobId> }`.
- Returns the source `{ jobId, attempt, queue }` of that row, or `null` if no source recorded the link.
- Uses the existing `record_terminal_detail_gin` index. Defensive `ORDER BY captured_at DESC LIMIT 1` in case duplicates ever exist.

SQL:

```sql
SELECT job_id AS "jobId", attempt, queue
FROM pgbossier.record
WHERE terminal_detail @> jsonb_build_object('deadLetteredAs', $1::text)
ORDER BY captured_at DESC
LIMIT 1;
```

The forensic flow: DLQ worker has `dlqJobId` â†’ `findDeadLetterSource(dlqJobId)` â†’ `{sourceJobId, attempt, sourceQueue}` â†’ `getRetryHistory(sourceJobId)` â†’ full retry chain.

### Forward reader (should-land in v1)

```typescript
client.findDeadLetterTarget(
  sourceJobId: string
): Promise<{ dlqJobId: string; attempt: number } | null>;
```

Behavior:
- Queries `pgbossier.record` for `sourceJobId`'s most recent `failed` row whose `terminal_detail.deadLetteredAs` is set.
- Returns `{ dlqJobId, attempt }` where `attempt` is the source's attempt that produced the DLQ link.
- Returns `null` if the source was never linked.

SQL:

```sql
SELECT terminal_detail->>'deadLetteredAs' AS "dlqJobId", attempt
FROM pgbossier.record
WHERE job_id = $1
  AND state = 'failed'
  AND terminal_detail ? 'deadLetteredAs'
ORDER BY attempt DESC
LIMIT 1;
```

### Composition with Goal 2

Goal 2's writer (`recordTerminalDetail`) is changed to use the same `COALESCE(...) || ...` JSONB merge pattern (decision 3). After the fix:

- Calling `recordTerminalDetail` first, then `recordDeadLetter`: both keys present in `terminal_detail`. Expected.
- Calling `recordDeadLetter` first, then `recordTerminalDetail`: both keys present in `terminal_detail`. The `deadLetteredAs` field survives the second call because it's not in the second call's payload, and JSONB `||` only overwrites keys that are in both objects.
- Calling `recordTerminalDetail` twice on the same row: keys merge, same-key overwrites. The most recent call wins per-key, which is the correct audit semantic.

The Goal 2 reader (`JobRecord.terminalDetail`) returns the merged shape. The discriminated-union types in `src/terminal-detail.ts` are extended with one optional field on `TerminalDetailFailed`:

```typescript
export type TerminalDetailFailed = {
  class: 'transient' | 'non_retryable';
  message?: string;
  where?: string;
  deadLetteredAs?: string;  // pg-bossier-managed; set by recordDeadLetter
} & Record<string, unknown>;
```

## What does NOT change

- `pgbossier.record` schema â€” no new column.
- The capture trigger â€” unchanged.
- `getRetryHistory(jobId)` â€” unchanged. For a DLQ job, returns the DLQ job's own retry chain only (the source's chain is reachable via `findDeadLetterSource` + `getRetryHistory` on the source's id).
- `recordTerminalDetail`'s public signature, validation rules, and state-binding behavior. Only its internal SQL changes (key-level merge instead of full-object overwrite). Goal 2's tests continue to assert key-level behavior.
- pg-boss surfaces touched â€” none new.
- **`progress` (Goal 6) data is NOT copied source â†’ DLQ.** The DLQ job starts with no progress; its chronicle entries are its own. The source job's `progress` history stays on the source's rows.
- **`boss.retry(dlqJobId)` (the SQS-redrive analogue) does not disturb the existing `deadLetteredAs` link.** The DLQ job gets a new attempt; its `pgbossier.record` row gets a new attempt row via the capture trigger; the existing `deadLetteredAs` on the *source* job's row is unaffected.

## Backfill story

**None required.** Existing installs continue to work. Pre-existing source rows have no `deadLetteredAs` in `terminal_detail`; `findDeadLetterSource` returns `null` for any DLQ id whose source predates writer adoption. Consumers opt in by setting `_originalJobId` on `boss.send()` and calling `recordDeadLetter` from their DLQ-handler code.

**Goal 2's writer change is in-place** (no migration) â€” the JSONB column shape is unchanged; only the writer's UPDATE SQL changes. Existing rows are unaffected.

## What this changes for descent-app

Today, if descent-app has a DLQ handler, it has to:

```js
// before Goal 3
const dlqJob = await boss.fetch('descent.dlq');
const sourceId = dlqJob.data._originalJobId;  // by convention
const history = await pool.query(
  `SELECT * FROM pgboss.job WHERE id = $1`, [sourceId]
);  // ...except that row may be gone because pg-boss deleted it
```

After Goal 3:

```js
// after Goal 3
const dlqJob = await boss.fetch('descent.dlq');
const sourceId = dlqJob.data._originalJobId;  // still the consumer's bookkeeping
await bossier.recordDeadLetter({ sourceJobId: sourceId, dlqJobId: dlqJob.id });

// ...later, in a forensic UI:
const source = await bossier.findDeadLetterSource(dlqJob.id);
// â†’ { jobId, attempt, queue }
const fullHistory = await bossier.getRetryHistory(source.jobId);
// fullHistory has every attempt, including inputs, outputs, terminal_detail
```

The shape that disappears: raw SQL against `pgboss.job` for a DLQ source. The shape that stays: descent-app's discipline of setting `_originalJobId` on `boss.send()`.

## Risks (named from the debate, mitigations included)

1. **Silent absence of the link.** Consumers who forget to call `recordDeadLetter` (or to set `_originalJobId`) get nothing â€” no error, no warning, no row. **Mitigation:** loud README + JSDoc that names this as the failure mode. Follow-up issue for an "expected DLQ links missing" ops diagnostic.

2. **Writer addresses the wrong attempt** under unusual lifecycles. `ORDER BY attempt DESC LIMIT 1` returns the most recent `failed` row; if the consumer somehow calls `recordDeadLetter` *before* the source's final failure has landed in `pgbossier.record`, the writer silently no-ops (no `failed` row matches the SELECT). **Mitigation:** the writer is a no-op when no `failed` row exists for `sourceJobId`; logged with `reason: 'not_found'`. The "call from DLQ-handler, which runs after the source is dead" ordering is the supported pattern.

3. **`recordDeadLetter` and `recordTerminalDetail` race** if both write to the same row's `terminal_detail` concurrently. **Mitigation:** the JSONB `||` merge in Postgres is atomic per UPDATE; both writers use it after the OPTION-A fix. Worst case is one of two transactions retries on serialization conflict (rare); no data loss.

4. **Conflicting `dlqJobId` on second call.** If a consumer bug calls `recordDeadLetter` twice with different `dlqJobId` values for the same source, the spec defines the second call as a no-op with a warning log (the first link wins). This prevents silent overwriting of an established link.

5. **`_originalJobId` consumer discipline.** Without this in `data`, the DLQ handler can't supply `sourceJobId`. **Mitigation:** named contract in README + JSDoc. Future ergonomic improvement: a `bossier.send()` wrapper that auto-injects.

## Test plan

Following the Goal 2 pattern, the implementation tests cover:

1. **Round-trip happy path.** Worker fails a job with `retryLimit: 0` configured to dead-letter into a separate queue. DLQ handler calls `recordDeadLetter`. `findDeadLetterSource(dlqJobId)` returns `{jobId, attempt, queue}`. `findDeadLetterTarget(sourceJobId)` returns `{dlqJobId, attempt}`. Reader narrowing exposes `terminal_detail.deadLetteredAs` typed.

2. **Composition with `recordTerminalDetail`.** Call sequence: `recordTerminalDetail({class: 'transient', message: 'rate limited'})` then `recordDeadLetter({sourceJobId, dlqJobId})`. Reader returns merged `{class: 'transient', message: 'rate limited', deadLetteredAs: '<id>'}`. Both writers' fields present.

3. **Reverse composition order.** `recordDeadLetter` first, then `recordTerminalDetail({class: 'non_retryable'})`. Reader returns `{class: 'non_retryable', deadLetteredAs: '<id>'}`. The `deadLetteredAs` survives the second call (this is the test the OPTION-A fix specifically enables).

4. **Source row not found â€” silent no-op.** Call `recordDeadLetter` with a `sourceJobId` that has no `pgbossier.record` row. Resolves without throwing. Warning logged with `reason: 'not_found'`.

5. **Conflicting second `recordDeadLetter` call.** Call with `sourceJobId=A, dlqJobId=X`, then call with `sourceJobId=A, dlqJobId=Y`. After second call, `terminal_detail.deadLetteredAs = X` (first wins). Warning logged.

6. **Idempotent same-id second call.** Call with `sourceJobId=A, dlqJobId=X` twice. After second call, `terminal_detail.deadLetteredAs = X`. No warning.

7. **Chained DLQ (A â†’ B â†’ C).** Set up sources A and B (where B is itself a DLQ job from A). Call `recordDeadLetter({sourceJobId: A, dlqJobId: B})` and `recordDeadLetter({sourceJobId: B, dlqJobId: C})`. `findDeadLetterSource(C)` returns B's row. `findDeadLetterSource(B)` returns A's row. Each call walks one hop.

8. **Goal 2 regression: key-level merge.** `recordTerminalDetail` called twice with overlapping keys â€” second call's value wins at the key level; non-overlapping keys from the first call survive. This locks in the OPTION-A semantic and replaces Goal 2's existing "full-object last-writer-wins" test (which is now incorrect against the new behavior).

9. **GIN index usage.** EXPLAIN ANALYZE confirms `findDeadLetterSource` uses `record_terminal_detail_gin`.
